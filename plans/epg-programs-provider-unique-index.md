## Bug (confirmed twice — DB + live)

`epg_programs` are stored **per provider** (`providerId` column; a source refresh writes a copy for each
linked provider — `DefaultEpgRefresher`: `activeProviderIds.forEach { importXmltv(providerId, source, …) }`).
But the UNIQUE index omits `providerId`:

```
index_epg_programs_epgSourceId_epgChannelId_stableKey  UNIQUE (epgSourceId, epgChannelId, stableKey)
```

`stableKey = hash(epgSourceStableKey : epgChannelId : startTime : endTime : title)` — **no providerId**
(`RoomEpgRepository.kt:458`); the program `id` **does** include providerId (`:448`). So when **one EPG
source is linked to ≥2 playlists**, provider A's import commits, then provider B's identical rows (same
`epgSourceId/epgChannelId/stableKey`, different `providerId/id`) hit the UNIQUE index → **SQLITE_CONSTRAINT
(2067)** → B's import throws → `EpgRefreshWorker` returns `retry` → WorkManager backoff **retry loop**
(re-downloads the XMLTV feed each attempt) and B gets **zero** EPG from those sources.

**Evidence (pulled emulator DB, 148 MB):**
- 3 playlists: `Test` (M3U), `Test #1` (M3U), `Test Xtream`. `Test` + `Test #1` are each linked to the
  **same** 3 sources (b24402f2 / e81d77fa / 34815d04).
- `epg_programs`: `Test` has all rows; **`Test #1` = 0 rows** (blocked). `Test Xtream` (own source) fine.
- Live logcat (5 min after restart): all 3 `PlaylistRefreshWorker` SUCCESS; `EpgRefreshWorker` = 2 SUCCESS
  / **3 RETRY** (the shared-source imports for the 2nd provider, looping).

**Not caused by the EPG-priority/logging work** — the unique index + per-provider import predate it;
sharing a source across playlists is the trigger (e.g. two M3U playlists both linked to the same guides).

## Fix

Add `providerId` to the unique index → `(providerId, epgSourceId, epgChannelId, stableKey)`. This aligns
the constraint with the per-provider `id`, so each provider's copy is unique on its own and coexists.

**Migration SQL (v19→v20):**
```sql
DROP INDEX IF EXISTS `index_epg_programs_epgSourceId_epgChannelId_stableKey`;
CREATE UNIQUE INDEX IF NOT EXISTS `index_epg_programs_providerId_epgSourceId_epgChannelId_stableKey`
    ON `epg_programs` (`providerId`, `epgSourceId`, `epgChannelId`, `stableKey`);
```

**Safe on existing data:** the old index guaranteed uniqueness on the sub-key `(epgSourceId, epgChannelId,
stableKey)`, so the super-key adding `providerId` can only be *more* unique → no duplicate can exist → the
new UNIQUE index builds without conflict. **Dry-run on the real 148 MB DB: 45857 rows, 0 dups on the new
key, DROP+CREATE in 0.179 s.** No data backfill: the blocked provider imports cleanly on its next EPG
refresh (the pending RETRY workers then succeed).

## Out of scope (note only)

- **Don't retry a deterministic constraint failure** in `EpgRefreshWorker` (return failure, not retry).
  Secondary — the index fix removes the trigger; a separate small hardening if we want it later.
- **ADR-002 "no per-provider EPG copies"** — the code *does* store per-provider (programs are keyed to the
  provider's `channelId`, since the same source maps to different provider channels). The index fix aligns
  the constraint with that real design. Moving to true single-copy storage is a large separate rework, not
  this fix.
