## 1. Problem & goal

**Symptom:** while a bulk import writes (EPG refresh, playlist/catalog refresh), any other DB write
hangs — group sort/hide/reorder, or **Save** on a playlist, spins until the import finishes.

**Root cause:** SQLite is **single-writer per database file** (WAL parallelises only *readers*). Both big
imports run as **one large `withTransaction`**, holding the single writer for the whole import:
- EPG: `RoomEpgRepository.importXmltv` — one `withTransaction` around delete + insert of 58k–200k
  programmes (`data/epg/.../RoomEpgRepository.kt:183`).
- Catalog: `RoomCatalogImportRepository.import*` — one `withTransaction` around all
  channels/movies/series/seasons/episodes/categories (`data/media/.../RoomCatalogImportRepository.kt:49,99,170`).

**Goal:** interactive writes never block on a background import; imports stay atomic (no partial/flicker
while browsing); scales as catalog/EPG grow. WAL is already on (`VivicastDatabaseFactory.kt:26`).

---

## 2. DECISION — keep String-PK, use separate staging tables + fingerprint delta-merge

Considered three mechanisms:
- **Chunked commits in-place** (no staging): tiny blocks but non-atomic (per-chunk re-render churn while
  browsing; partial snapshots).
- **Rowid PK + negative-id staging** (StreamVault EPG trick): needs an autogenerated `Long` PK so the
  staging discriminator (`providerId`) is a swappable column. **Rejected — see below.**
- **Separate staging tables + fingerprint delta-merge** (StreamVault *catalog* pattern): PK-scheme
  agnostic, atomic, works with our String PKs. **Chosen.**

### Why NOT rowid / negative-id (decisive for our codebase)
1. **Our FTS is trigger-based on the live tables.** `search_channels_fts` / `_movies_` / `_series_` /
   `_epg_fts` are maintained by SQL triggers `AFTER INSERT/UPDATE/DELETE` on channels/movies/series/
   epg_programs (`VivicastMigrations.kt:557-677`). Negative-id staging writes staged rows *into the live
   table* → those triggers fire → staged rows pollute the search index; and the swap fires update
   triggers en masse. StreamVault avoids this only because they don't use triggers (they rebuild FTS,
   O(N)). Adopting rowid means abandoning our clean incremental trigger-FTS.
2. **Deterministic String ids are an asset for future features.** `id = "$providerId:channel:$hash(remoteId)"`
   is stable across refreshes. Any future feature that references content (reminders on a programme,
   cross-provider dedup, cloud sync, sharing, recommendations) uses that stable id directly. Rowids
   change every refresh → every such feature must indirect through `stableKey` — a permanent tax on
   future work. Adaptability argues *for* our scheme.
3. **We don't need rowid to hit the goal.** Separate staging tables give atomic, non-blocking imports
   with our String PKs, leaving the live schema, every read query, every cross-table reference, and every
   feature untouched — least risk, least rework.

### What we take / reject from StreamVault
Take: the **catalog staging pattern** (separate `*_stage` tables + fingerprint delta-merge), applied
uniformly to catalog **and** EPG. Reject: their **rowid/negative-id EPG trick** (exists only because of
their schema; would break our trigger-FTS and hurt adaptability).

### Honest trade-off (accepted)
A rare EPG *full replace* (user changes `timeShiftMinutes`, or first import) makes the delta ≈ the whole
table, so the merge transaction writes many rows and briefly blocks. That is **not** the
"background-refresh-blocks-my-edit" scenario — a periodic refresh has a small delta (the next 7 days
mostly overlap the previous), so its merge is tiny and effectively non-blocking. The rare full replace
happens on first import / an explicit setting change, where a brief wait is fine.

## 3. Architecture — staged + fingerprint delta-merge (uniform)

For each big import:
1. **Parse + stage (chunked, OUTSIDE the final txn):** build entity rows with the **real providerId**
   (deterministic ids come out final) + a `syncFingerprint`; insert into a parallel `*_stage` table via a
   shared chunked-commit helper. The single writer is released between chunks → interactive writes
   interleave. Staging tables carry **no FTS triggers** → no index pollution, no trigger churn.
2. **Merge (ONE short final txn)** per entity, three correlated-subquery steps:
   - `updateChangedFromStage` — update live rows whose `syncFingerprint != stage.syncFingerprint`
     (content columns only; never touch user-state columns).
   - `insertMissingFromStage` — `INSERT … SELECT … WHERE NOT EXISTS (live match by id)`.
   - `deleteStaleForStage` — `DELETE FROM live WHERE providerId = ? AND NOT EXISTS (stage match)`, plus
     the cross-table cascades for removed rows.
   Writing into the **live** tables here fires the existing FTS triggers on exactly the delta →
   incremental, correct. Room emits **one** invalidation per table → the UI flips old-complete →
   new-complete, no partial snapshot, no per-chunk churn.
3. **Cleanup:** clear `*_stage` for the session in a `finally`.

Small/moderate writers stay **direct inside the final merge txn** (they don't cause long blocks):
categories (+ D10 user-state preserve), seasons, epg_channels, epg_channel_mappings.

## 5. Shared chunked-commit helper
```kotlin
// Each chunk commits in its own transaction, yielding so the single writer is released and interactive
// writes (+ cancellation) interleave. ponytail: fixed chunk size; tune per table if needed.
suspend fun <T> List<T>.forEachChunkedTransaction(db: VivicastDatabase, size: Int, block: suspend (List<T>) -> Unit) {
    for (chunk in chunked(size)) { db.withTransaction { block(chunk) }; kotlinx.coroutines.yield() }
}
```
Chunk size **1_000–2_000** for stage inserts (down from the single-txn `EPG_IMPORT_CHUNK_SIZE = 20_000`).

## 6. Fingerprint
`syncFingerprint = sha256Hex(contentColumnsOnly)` — for a channel: name, channelNumber, logoUrl,
epgChannelId, categoryId, catchup flags. Excludes `id` (deterministic anyway), `createdAt/updatedAt`, and
user-state (there is none on channels; categories keep their own preserve-merge). The merge's
`updateChangedFromStage` compares `live.syncFingerprint != stage.syncFingerprint` so unchanged rows are
never rewritten → merge cost ∝ change, not catalog size.

## 7. Concurrency / crash / consistency
- **WAL:** already on — keep.
- **Session:** one in-flight import per (provider) / (source), guarded by a `Mutex`; `*_stage` cleared at
  start + in `finally`. If we ever allow concurrent same-provider imports, add a `sessionId` column
  (StreamVault `SyncCatalogStore` uses one) — not needed for v1.
- **Crash safety:** staging inserts commit to stage tables (invisible to reads). A crash before the merge
  leaves live untouched; startup cleanup drops orphaned stage rows. The merge itself is one atomic txn.
- **Read consistency:** all observe queries read live tables only; staging is separate → invisible. One
  merge txn → one Room invalidation → atomic old→new flip.
- **Ordering:** catalog import must precede EPG (mappings map over imported channels —
  `RoomEpgRepository.kt:187`); worker already enforces this (`RefreshExecution.kt:77-79`). Unchanged.

(or downgraded to a soft "wird aktualisiert…" hint). The refresh-button cross-lock has a separate
rationale (playlist→EPG auto-chain cascade) — keep it. Removing the gate is a follow-up once this lands
(touches visible UI → owner call).

## 12. Risks / open items
1. **EPG full-replace** (time-shift change / first import): delta ≈ all → merge writes many rows in one
   txn → brief block. Accepted (not the periodic-refresh scenario). If ever a problem, chunk the
   insert-missing step and accept a one-time non-atomic first import.
2. **Fingerprint CPU:** SHA-256 per row per import (~1s for 200k programmes on TV). Cheaper than the I/O
   it saves (unchanged rows not rewritten). Measure on a large Xtream provider.
3. **`removedIds`/delete-stale after the FULL staged set** — a per-chunk delete would wipe not-yet-staged
   rows. Staging tables provide the full set at merge time (this is *why* separate staging, not in-place
   per-chunk delete). The single most dangerous spot; covered by design + test.
4. **`IN (:ids)` 999-var limit** in cascades — chunk them.
5. **Season staging kept direct** — fine unless a provider has enormous season counts; revisit then.
